# Chapter IV: Methodology

This chapter presents the methodology employed in the study. It covers the research design, respondents of the study, data gathering instruments, data gathering procedures, and statistical tools. These components form a systematic approach to dataset development, transformer model fine-tuning, and performance analysis across varying code-switching densities.

---

## Research Design

The research uses a **mixed method** combining both **developmental** and **experimental** approaches.

### Developmental Approach

This research is classified as developmental since it involves:
- Developing a purpose-built dataset of annotated Taglish social media posts
- Developing a set of fine-tuned sentiment classification models based on transformer architectures

Developmental research is appropriate when the primary aim is to create a computational model or system to address a specific problem.

### Experimental Approach

This research is also experimental, as it involves manipulating several factors to examine how each influences model performance:

| Variable Type | Variables |
|---------------|-----------|
| **Independent** | Model architecture (mBERT, XLM-RoBERTa, TLUnified-RoBERTa) |
| | Training procedure (standard fine-tuning, QLoRA, QLoRA + SPCL) |
| | Code-switch density level |
| **Dependent** | Macro-F1 score |
| | Accuracy |
| | Precision |
| | Recall |

---

### Development Framework: Agile-MLOps Hybrid SDLC

To ensure systematic development and reproducibility, the study is guided by a **hybrid Software Development Life Cycle (SDLC)** framework that integrates **Agile principles** with an **MLOps-oriented lifecycle**.

- **Agile approach** — enables iterative experimentation through incremental development cycles, where dataset refinement, model fine-tuning, and evaluation are conducted in successive phases.
- **MLOps lifecycle** — structures the end-to-end pipeline into distinct stages with version control over datasets, hyperparameters, and trained models, as well as tracking experimental results to ensure traceability and comparability across iterations.

The study also adopts a **modular, API-oriented system design perspective**, where trained models are treated as deployable components that can be integrated into web-based applications. Deployment considerations include:
- Inference efficiency
- Model size optimization through quantization
- Scalability

**Agile-MLOps Lifecycle Phases:**

| Phase | Description |
|-------|-------------|
| 1. Data Collection & Preprocessing | Taglish datasets are curated and annotated |
| 2. Model Development | Transformer models are fine-tuned using different strategies |
| 3. Evaluation | Performance metrics are computed |
| 4. Iteration | Model improvements are applied based on results |
| 5. Deployment Preparation | Models are optimized for real-world integration |

## Respondents of the Study

The human evaluators in this study are **Bachelor of Science in Computer Science and Information Technology students from Partido State University**, recruited to assess the sentiment predictions generated by the best-performing model. These respondents are considered suitable evaluators as they are:

- Familiar with online social networking practices
- Knowledgeable of the language used in Taglish-based social media
- Capable of assessing sentiment expressions in an informal multilingual setting

**Total respondents:** 80 student evaluators

**Sampling strategy:** Convenience sampling — subjects are chosen according to their proximity, availability, and willingness to join the investigation process during the data collection period. This strategy is appropriate given that respondents are readily accessible to the researchers within the duration of the data collection period.

Respondents will rate the outputs generated by the most effective transformer model through a **researcher-developed Likert-scale questionnaire**.

## Data Gathering Instruments

### Dataset Composition

The dataset to be used consists of **7,707 manually curated samples** of Taglish social media posts collected from:
- Reddit
- YouTube
- Facebook
- AI-generated samples (for controlled data augmentation)

**Sentiment distribution:**

| Sentiment Class | Proportion |
|----------------|------------|
| Positive | ~33% |
| Negative | ~33% |
| Neutral | ~33% |

**Code-switch density annotation bands:**

| Band | English Token Proportion |
|------|--------------------------|
| Band 1 | 0–20% |
| Band 2 | 21–40% |
| Band 3 | 41–60% |
| Band 4 | 61–80% |
| Band 5 | 81–100% |

---

### Technical Performance Instruments

The primary instruments for technical assessment are **performance benchmarks**:

| Metric | Purpose |
|--------|---------|
| **Macro-F1 Score** (primary metric) | Accounts for class imbalance across the three sentiment classes |
| **Accuracy** | Overall proportion of correct predictions |
| **Precision** | Correctness of positive predictions per class |
| **Recall** | Coverage of actual positives per class |
| **Confusion Matrices** | Visualizes misclassification patterns among classes and per density band |
| **Per-density-band Macro-F1** | Analyzes model performance within each code-switch density band (Objective 3) |

---

### Failure Analysis Checklist

To address **Objective 4**, a **Failure Analysis Checklist** will be developed to categorize misclassified samples into the following error types:

| Error Category | Description |
|----------------|-------------|
| Sarcasm / Irony | Posts where expressed sentiment contradicts literal meaning |
| *Baliktad na salita* | Reversed Filipino slang (e.g., *'erpat'* for *'father'*, *'lodi'* for *'idol'*) |
| Netspeak | Text speak and informal internet expressions (e.g., *'marites'*, *'sana all'*) |
| Emoji-based posts | Posts where emojis are primary sentiment carriers |

Misclassified samples will be extracted using specific keywords and examined from model outputs saved as a **CSV file**.

---

### Likert-Scale Questionnaire

To address **Objective 5**, a **researcher-developed Likert-scale questionnaire** will assess respondents' perceptions of the accuracy, relevance, and usefulness of the model's sentiment predictions on sample Taglish social media posts.

**Scale gradations:**

| Score | Label |
|-------|-------|
| 5 | Strongly Agree |
| 4 | Agree |
| 3 | Neutral |
| 2 | Disagree |
| 1 | Strongly Disagree |

## Data Gathering Procedures

The data gathering procedures follow two parallel processes:
1. **Technical model development and benchmarking**
2. **Respondent evaluation**

---

### Step 1: Data Preprocessing

Prior to model training, the dataset will undergo preprocessing to ensure data quality and consistency:

- Removing noise — URLs, mentions, and unnecessary special characters
- Text normalization
- **Emoji preservation** — emojis are retained as they contribute to sentiment expression
- Tokenization using the respective tokenizer of each transformer model

**Dataset split ratio:**

| Split | Proportion | Purpose |
|-------|-----------|---------|
| Training | 80% | Model learning |
| Validation | 10% | Performance monitoring, early stopping |
| Test | 10% | Final unbiased evaluation |

---

### Step 2: Technical Evaluation

Models will be assessed using macro-F1 score, accuracy, precision, recall, and confusion matrices computed on the test dataset.

- **Macro-F1** is the primary metric — accounts for class balance across positive, negative, and neutral sentiments
- **Confusion matrices** examine misclassification patterns across sentiment classes and across all five code-switching density bands

---

### Step 3: Density-Based Curriculum Learning (SPCL)

The **Density-Primary Self-Paced Curriculum Learning (SPCL)** technique will be applied to the training data to gradually expose the model to increasingly complex code-switched structures.

By organizing training data according to density band and sample confidence, a **progressive learning approach** is achieved — beginning with purely monolingual text and advancing to balanced Taglish.

**Staged learning rate schedule:**

| Stage | Training Content | Learning Rate |
|-------|-----------------|---------------|
| Stage 1 | Pure monolingual posts (0–20%) | 3×10⁻⁵ |
| Stage 2 | Light code-switching (21–40%) | 2×10⁻⁵ |
| Stage 3 | Heavy code-switching (61–80%) | 1×10⁻⁵ |
| Stage 4 | Balanced Taglish (41–60%) | 8×10⁻⁶ |

**Additional training parameters:**
- Maximum sequence length: **128 tokens**
- Batch size and total training epochs will be finalized through preliminary experimentation on the validation set
- Each stage is subject to **early stopping** based on macro-F1 performance on the validation set, with:
  - Patience of **3 consecutive evaluations**
  - Mandatory warmup window of **2 evaluations per stage** to allow model stabilization following each distribution shift

**For QLoRA-based training:**
- Low-bit quantization applied to reduce computational requirements while maintaining model performance
- Best-performing model selected based on the highest macro-F1 score on the validation dataset

---

### Step 4: Qualitative Failure Analysis

A **failure analysis** will be conducted to identify error patterns in model misclassifications. Researchers will manually analyze misclassified outputs using the Failure Analysis Checklist (four categories: sarcasm/irony, *baliktad na salita*, netspeak, and emoji-based posts). Misclassified outputs will be gathered in a **CSV file** for analysis.

---

### Step 5: Respondent Evaluation

Respondents' perceptions of model performance will be determined through the researcher-made Likert-scale questionnaire (1 = Strongly Disagree to 5 = Strongly Agree). Before use, the questionnaire will be subjected to **content validity** to verify its clarity and relevance to the task.

## Statistical Tools

This study employs two categories of statistical tools corresponding to its two main evaluation components.

---

### Machine Learning Model Evaluation

| Tool | Purpose |
|------|---------|
| **Macro-averaged F1 Score (Macro-F1)** | Primary metric; calculates average F1 score for all classes regardless of occurrence frequency. Appropriate because class distribution may be uneven despite efforts to balance the three sentiment classes. |
| **Accuracy** | Overall correctness of predictions |
| **Precision** | Per-class correctness of positive predictions |
| **Recall** | Per-class coverage of actual positives |
| **Confusion Matrix** | Displays the number of correct and incorrect predictions for each of the three classes |
| **Per-density-band Macro-F1** | Calculated for each of the five code-switch density bands to analyze how increased code-switching affects model performance |

---

### Human Respondent Data Analysis

| Tool | Purpose |
|------|---------|
| **Weighted Mean** | Calculates the average rating per questionnaire item, providing insight into how respondents perceive the usefulness and accuracy of model outputs |
| **Frequency and Percentage** | Calculates the distribution of respondents' answers for each questionnaire item |

These statistical tools are appropriate given that responses to Likert-scale items constitute **ordinal data**.
